Git Commands Guide for Beginners

This guide will help you understand Git commands and their usage in a team environment.

1. Initial Setup

1.1 Configure Git

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
These commands set up your identity for all Git repositories on your system.

2. Starting a New Project

2.1 Initialize a New Repository

git init

Example:

mkdir my-project

cd my-project

git init

2.2 Clone an Existing Repository

git clone [repository-url]

Example:

git clone https://github.com/username/project-name.git

3. Basic Workflow

3.1 Check Status

git status
Always check status before making changes to see what files are modified.

3.2 Add Files to Staging

git add [file-name]
git add .

Example:

git add index.html

git add . (to add all changed files)

3.3 Commit Changes

git commit -m "Your commit message"
Write clear, descriptive commit messages that explain what changes were made.

4. Working with Branches

4.1 Create a New Branch

git branch [branch-name]
git checkout -b [branch-name]

Example:

git checkout -b feature/login-page

4.2 Switch Between Branches

git checkout [branch-name]
Always commit or stash your changes before switching branches.

4.3 List All Branches

git branch
git branch -a

5. Team Collaboration

5.1 Push Changes to Remote

git push origin [branch-name]

Example:

git push origin feature/login-page

5.2 Pull Latest Changes

git pull origin [branch-name]
Always pull latest changes before starting work on a new feature.

5.3 Resolve Merge Conflicts

When conflicts occur:
  1. Open the conflicted files
  2. Look for conflict markers (<<<<<<<, =======, >>>>>>>)
  3. Choose which changes to keep
  4. Remove conflict markers
  5. Add and commit the resolved files

6. Common Issues and Solutions

6.1 Undo Local Changes

git checkout -- [file-name]
This will discard changes in the working directory.

6.2 Stash Changes

git stash
git stash pop

Example:

git stash (to save changes temporarily)

git stash pop (to restore saved changes)

6.3 View Commit History

git log
git log --oneline

7. Best Practices

  • Always create a new branch for each feature or bug fix
  • Write clear, descriptive commit messages
  • Pull latest changes before starting work
  • Test your changes before pushing
  • Keep your branches up to date with the main branch
  • Delete branches after merging them

8. Common Team Workflow

  1. Pull latest changes: git pull origin main
  2. Create new branch: git checkout -b feature/your-feature
  3. Make changes and commit: git add . and git commit -m "message"
  4. Push changes: git push origin feature/your-feature
  5. Create Pull Request on GitHub/GitLab
  6. After approval, merge to main branch

Remember:

Git is a powerful tool that helps teams work together efficiently. Take time to understand these commands and practice them regularly. When in doubt, always check the status of your repository before making changes.

9. टीम प्रोजेक्ट में Git का उपयोग (Team Project Guide in Hindi)

यह गाइड विशेष रूप से टीम प्रोजेक्ट्स के लिए है, जहां कई डेवलपर्स एक साथ काम कर रहे हैं।

9.1 प्रोजेक्ट सेटअप

git config --global user.name "आपका नाम"
git config --global user.email "आपका.ईमेल@example.com"
सबसे पहले अपनी पहचान सेट करें। यह जरूरी है ताकि टीम को पता चले कि कौन से चेंजेस किसने किए।

9.2 डेली वर्कफ्लो

सुबह का काम:

git checkout main
git pull origin main

हर दिन की शुरुआत मेन ब्रांच को अपडेट करके करें

नए फीचर पर काम शुरू करना:

git checkout -b feature/your-feature-name

उदाहरण: git checkout -b feature/login-page

9.3 कोड चेंजेस का प्रबंधन

छोटे-छोटे कमिट्स करें:

git add .
git commit -m "लॉगिन फॉर्म वैलिडेशन जोड़ा"
git push origin feature/your-feature-name

9.4 टीम के साथ सिंक रहना

git checkout main
git pull origin main
git checkout feature/your-feature-name
git merge main

रेगुलर इंटरवल पर मेन ब्रांच से अपडेट लेते रहें

9.5 महत्वपूर्ण टिप्स

  • हमेशा नई फीचर के लिए नई ब्रांच बनाएं
  • कमिट मैसेज हिंदी या अंग्रेजी में स्पष्ट लिखें
  • पुश करने से पहले लोकल टेस्टिंग करें
  • टीम के साथ रेगुलर कम्युनिकेशन रखें
  • कॉन्फ्लिक्ट्स को तुरंत रिजॉल्व करें

9.6 कॉन्फ्लिक्ट रिजॉल्यूशन

कॉन्फ्लिक्ट आने पर:

  1. कॉन्फ्लिक्ट वाली फाइल्स चेक करें
  2. टीम से डिस्कस करें
  3. सही कोड रखें और कॉन्फ्लिक्ट मार्कर्स हटाएं
  4. चेंजेस को कमिट करें

9.7 सुरक्षा टिप्स

  • कभी भी सीधे मेन ब्रांच पर काम न करें
  • पासवर्ड या API कीज़ कभी कमिट न करें
  • रेगुलर बैकअप लेते रहें
  • अनजान कमांड्स का उपयोग न करें

याद रखें:

टीम प्रोजेक्ट में Git का सफल उपयोग कम्युनिकेशन, ऑर्गनाइजेशन और डिसिप्लिन पर निर्भर करता है। रेगुलर प्रैक्टिस से आप और आपकी टीम बेहतर तरीके से काम कर सकेंगे।

  • कमिट मैसेज में क्या चेंज किया है वो स्पष्ट लिखें

11. Git Cherry-Pick (विस्तृत गाइड)

Git cherry-pick एक शक्तिशाली कमांड है जो आपको किसी एक कमिट को एक ब्रांच से दूसरी ब्रांच में ले जाने की अनुमति देता है।

11.1 Cherry-Pick क्या है?

Cherry-pick का मतलब है किसी एक कमिट को चुनकर उसे दूसरी ब्रांच में ले जाना। यह तब उपयोगी होता है जब:

  • आप किसी एक फीचर को एक ब्रांच से दूसरी ब्रांच में ले जाना चाहते हैं
  • किसी बग फिक्स को एक ब्रांच से दूसरी ब्रांच में ले जाना चाहते हैं
  • किसी एक कमिट को मर्ज करना चाहते हैं, पूरी ब्रांच को नहीं

11.2 Cherry-Pick का उपयोग कैसे करें?

बेसिक सिंटैक्स:

git cherry-pick <commit-hash>

उदाहरण:

मान लीजिए आपके पास दो ब्रांच हैं:

  • main (मुख्य ब्रांच)
  • feature/login (लॉगिन फीचर ब्रांच)

आप feature/login ब्रांच में एक बग फिक्स किया है और उसे main ब्रांच में ले जाना चाहते हैं:

# पहले feature/login ब्रांच में जाएं
git checkout feature/login

# कमिट हिस्ट्री देखें
git log --oneline

# मान लीजिए बग फिक्स का कमिट हैश है: abc1234
# main ब्रांच पर स्विच करें
git checkout main

# कमिट को cherry-pick करें
git cherry-pick abc1234

11.3 Cherry-Pick के विकल्प

1. कमिट को एडिट करके cherry-pick करना:

git cherry-pick -e abc1234

2. कमिट मैसेज को बदले बिना cherry-pick करना:

git cherry-pick -x abc1234

3. कई कमिट्स को एक साथ cherry-pick करना:

git cherry-pick abc1234 def5678

11.4 Cherry-Pick केस स्टडी

सीनारियो: आपके पास एक वेबसाइट प्रोजेक्ट है जिसमें दो ब्रांच हैं:

  • main (प्रोडक्शन कोड)
  • staging (टेस्टिंग के लिए)

समस्या: staging ब्रांच में एक सुरक्षा फिक्स किया गया है जिसे तुरंत main ब्रांच में ले जाना है।

समाधान:

# staging ब्रांच में जाएं
git checkout staging

# कमिट हिस्ट्री देखें
git log --oneline

# सुरक्षा फिक्स का कमिट हैश है: 789xyz
# main ब्रांच पर स्विच करें
git checkout main

# कमिट को cherry-pick करें
git cherry-pick 789xyz

# चेंजेस को रिमोट में पुश करें
git push origin main

11.5 Cherry-Pick के फायदे

  • स्पेसिफिक चेंजेस को टारगेट कर सकते हैं
  • पूरी ब्रांच को मर्ज किए बिना एक कमिट ले सकते हैं
  • क्विक फिक्स को एक ब्रांच से दूसरी ब्रांच में ले जा सकते हैं
  • फ्लेक्सिबल वर्कफ्लो बनाता है

11.6 Cherry-Pick के नुकसान

  • कॉन्फ्लिक्ट्स हो सकते हैं
  • कमिट हिस्ट्री कॉम्प्लेक्स हो सकती है
  • डिपेंडेंसी इश्यूज हो सकते हैं
  • टेस्टिंग जरूरी है

11.7 बेस्ट प्रैक्टिसेज

  • हमेशा पहले टेस्ट करें
  • कॉन्फ्लिक्ट्स को तुरंत रिजॉल्व करें
  • क्लियर कमिट मैसेज रखें
  • डिपेंडेंसीज को ध्यान में रखें
  • डॉक्युमेंटेशन अपडेट करें

Remember:

Git is a powerful tool that helps teams work together efficiently. Take time to understand these commands and practice them regularly. When in doubt, always check the status of your repository before making changes.

12. महत्वपूर्ण Git कमांड्स (Important Git Commands)

यह सेक्शन डेवलपर्स द्वारा सबसे ज्यादा उपयोग किए जाने वाले Git कमांड्स को विस्तार से समझाता है।

12.1 Git Merge

क्या है Git Merge?

Git merge दो ब्रांचों को एक साथ जोड़ने का काम करता है। यह एक ब्रांच के सभी कमिट्स को दूसरी ब्रांच में ले आता है।

बेसिक सिंटैक्स:

git merge <branch-name>

उदाहरण:

# मान लीजिए आप feature/login ब्रांच को main में मर्ज करना चाहते हैं
git checkout main
git merge feature/login

महत्वपूर्ण विकल्प:

# नो-फास्ट-फॉरवर्ड मर्ज (हमेशा नया कमिट बनाता है)
git merge --no-ff feature/login

# मर्ज कैंसल करना
git merge --abort

12.2 Git Pull

क्या है Git Pull?

Git pull रिमोट रिपॉजिटरी से लेटेस्ट चेंजेस को लोकल रिपॉजिटरी में लाता है। यह git fetch और git merge का कॉम्बिनेशन है।

बेसिक सिंटैक्स:

git pull <remote> <branch>

उदाहरण:

# मेन ब्रांच से लेटेस्ट चेंजेस लाना
git pull origin main

# स्पेसिफिक ब्रांच से चेंजेस लाना
git pull origin feature/login

महत्वपूर्ण विकल्प:

# रिबेस के साथ पुल करना
git pull --rebase origin main

# फोर्स पुल (सावधानी से उपयोग करें)
git pull --force origin main

12.3 Git Push

क्या है Git Push?

Git push लोकल कमिट्स को रिमोट रिपॉजिटरी में अपलोड करता है।

बेसिक सिंटैक्स:

git push <remote> <branch>

उदाहरण:

# मेन ब्रांच में चेंजेस पुश करना
git push origin main

# नई ब्रांच पुश करना
git push -u origin feature/login

महत्वपूर्ण विकल्प:

# फोर्स पुश (सावधानी से उपयोग करें)
git push --force origin main

# टैग्स पुश करना
git push origin --tags

12.4 Git Stash

क्या है Git Stash?

Git stash अस्थायी रूप से आपके बदलावों को स्टोर करता है ताकि आप बिना कमिट किए दूसरी ब्रांच पर स्विच कर सकें।

बेसिक सिंटैक्स:

# बदलावों को स्टैश करना
git stash

# स्टैश किए गए बदलावों को वापस लाना
git stash pop

# स्टैश लिस्ट देखना
git stash list

उदाहरण:

# मान लीजिए आप feature/login पर काम कर रहे हैं
# और तुरंत main ब्रांच पर एक बग फिक्स करना है

# पहले करंट चेंजेस को स्टैश करें
git stash

# main ब्रांच पर स्विच करें
git checkout main

# बग फिक्स करें और कमिट करें
git add .
git commit -m "Fixed critical bug"

# वापस feature/login पर जाएं
git checkout feature/login

# स्टैश किए गए चेंजेस को वापस लाएं
git stash pop

महत्वपूर्ण विकल्प:

# स्टैश को नाम देना
git stash save "login-page-changes"

# स्पेसिफिक स्टैश को एप्लाई करना
git stash apply stash@{2}

# स्टैश को डिलीट करना
git stash drop stash@{0}

12.5 अन्य महत्वपूर्ण कमांड्स

1. Git Reset

# लास्ट कमिट को अनडू करना
git reset --soft HEAD~1

# स्टेजिंग एरिया को क्लियर करना
git reset HEAD

# वर्किंग डायरेक्टरी को क्लियर करना
git reset --hard HEAD

2. Git Revert

# किसी कमिट को रिवर्ट करना
git revert <commit-hash>

3. Git Rebase

# इंटरैक्टिव रीबेस
git rebase -i HEAD~3

4. Git Log

# ग्राफिकल लॉग
git log --graph --oneline --all

# स्पेसिफिक फाइल का हिस्ट्री
git log -p filename

12.6 बेस्ट प्रैक्टिसेज

  • हमेशा पुल करने से पहले स्टैश करें
  • फोर्स पुश/पुल का उपयोग सावधानी से करें
  • क्लियर कमिट मैसेज लिखें
  • रेगुलर बैकअप लें
  • कॉन्फ्लिक्ट्स को तुरंत रिजॉल्व करें

Remember:

Git is a powerful tool that helps teams work together efficiently. Take time to understand these commands and practice them regularly. When in doubt, always check the status of your repository before making changes.

13. Git बेस्ट प्रैक्टिसेज (विस्तृत गाइड)

यह सेक्शन Git के सबसे महत्वपूर्ण बेस्ट प्रैक्टिसेज को विस्तार से समझाता है।

13.1 पुल करने से पहले स्टैश करें

क्यों जरूरी है?

जब आप किसी ब्रांच पर काम कर रहे होते हैं और दूसरी ब्रांच से चेंजेस लाना चाहते हैं, तो आपके बिना कमिट किए गए चेंजेस कॉन्फ्लिक्ट कर सकते हैं।

कैसे करें?

# पहले अपने चेंजेस को स्टैश करें
git stash

# अब सुरक्षित रूप से पुल करें
git pull origin main

# स्टैश किए गए चेंजेस को वापस लाएं
git stash pop

उदाहरण:

मान लीजिए आप feature/login ब्रांच पर काम कर रहे हैं और main ब्रांच से नए चेंजेस लाने हैं:

# पहले चेंजेस को स्टैश करें
git stash save "login-page-changes"

# मेन ब्रांच से चेंजेस लाएं
git pull origin main

# स्टैश किए गए चेंजेस को वापस लाएं
git stash pop

13.2 फोर्स पुश/पुल का सावधानी से उपयोग

क्या है फोर्स पुश/पुल?

फोर्स पुश/पुल Git हिस्ट्री को ओवरराइड कर देता है, जो खतरनाक हो सकता है।

कब उपयोग करें?

  • जब आप पूरी तरह से सुनिश्चित हों कि आपका कोड सही है
  • जब आपको पता हो कि इससे किसी और का काम प्रभावित नहीं होगा
  • जब आपके पास कोई और विकल्प न हो

सुरक्षित तरीका:

# पहले बैकअप बनाएं
git branch backup-branch

# फिर फोर्स पुश करें
git push --force origin main

13.3 क्लियर कमिट मैसेज लिखें

अच्छे कमिट मैसेज की विशेषताएं:

  • संक्षिप्त और स्पष्ट
  • क्या बदलाव किया है, यह बताता हो
  • क्यों बदलाव किया है, यह बताता हो
  • टाइपो-फ्री हो

उदाहरण:

# बुरा उदाहरण
git commit -m "fixed bug"

# अच्छा उदाहरण
git commit -m "Fix: Login form validation not working on mobile devices"

# बेहतर उदाहरण
git commit -m "Fix: Login form validation not working on mobile devices\n\n- Added mobile-specific validation rules\n- Fixed touch event handling\n- Updated error messages for mobile users"

13.4 रेगुलर बैकअप लें

बैकअप के तरीके:

  1. ब्रांच बैकअप:
    # नई बैकअप ब्रांच बनाएं
    git branch backup-2024-04-07

    # बैकअप को रिमोट में पुश करें
    git push origin backup-2024-04-07
  2. टैग बैकअप:
    # वर्जन टैग बनाएं
    git tag v1.0.0-backup

    # टैग को रिमोट में पुश करें
    git push origin v1.0.0-backup

बैकअप शेड्यूल:

  • हर दिन काम खत्म करने से पहले
  • मेजर चेंजेस के बाद
  • रिलीज से पहले

13.5 कॉन्फ्लिक्ट्स को तुरंत रिजॉल्व करें

कॉन्फ्लिक्ट रिजॉल्यूशन स्टेप्स:

  1. कॉन्फ्लिक्ट को पहचानें:
    git status
  2. कॉन्फ्लिक्ट वाली फाइल्स को एडिट करें:
    # कॉन्फ्लिक्ट मार्कर्स को देखें
    <<<<<<< HEAD
    your changes
    =======
    their changes
    >>>>>>> branch-name
  3. चेंजेस को स्टेज करें:
    git add conflicted-file.txt
  4. कमिट करें:
    git commit -m "Resolve merge conflicts in login form"

कॉन्फ्लिक्ट रिजॉल्यूशन टिप्स:

  • कॉन्फ्लिक्ट आने पर तुरंत एक्शन लें
  • टीम के साथ कम्युनिकेट करें
  • दोनों पक्षों के चेंजेस को समझें
  • टेस्ट करें कि सब कुछ ठीक काम कर रहा है

याद रखें:

ये बेस्ट प्रैक्टिसेज आपके और आपकी टीम के काम को सुरक्षित और व्यवस्थित रखने में मदद करेंगे। इन्हें नियमित रूप से फॉलो करें और टीम के साथ शेयर करें।

14. Git कॉन्फिगरेशन फाइल्स (विस्तृत गाइड)

यह सेक्शन Git की महत्वपूर्ण कॉन्फिगरेशन फाइल्स के बारे में विस्तृत जानकारी प्रदान करता है।

14.1 .gitignore फाइल

क्या है .gitignore?

.gitignore एक टेक्स्ट फाइल है जो Git को बताती है कि किन फाइल्स और फोल्डर्स को इग्नोर करना है।

महत्वपूर्ण उपयोग:

  • सिस्टम फाइल्स:
    # macOS सिस्टम फाइल्स
    .DS_Store

    # Windows सिस्टम फाइल्स
    Thumbs.db
  • डिपेंडेंसी फोल्डर्स:
    # Node.js
    node_modules/

    # Python
    __pycache__/
    venv/
  • एनवायरनमेंट फाइल्स:
    # एनवायरनमेंट वेरिएबल्स
    .env
    .env.local

    # कॉन्फिगरेशन फाइल्स
    config.json

ग्लोबल .gitignore:

# ग्लोबल .gitignore सेट करें
git config --global core.excludesfile ~/.gitignore_global

14.2 .gitattributes फाइल

क्या है .gitattributes?

.gitattributes फाइल Git को बताती है कि विभिन्न फाइल टाइप्स के साथ कैसे व्यवहार करना है।

महत्वपूर्ण उपयोग:

# लाइन एंडिंग्स को नॉर्मलाइज करें
* text=auto

# बाइनरी फाइल्स को मार्क करें
*.png binary
*.jpg binary

# प्रोग्रामिंग लैंग्वेज स्पेसिफिक सेटिंग्स
*.js text eol=lf
*.py text eol=lf

14.3 .gitconfig फाइल

क्या है .gitconfig?

.gitconfig फाइल Git के लिए यूजर-स्पेसिफिक कॉन्फिगरेशन सेटिंग्स स्टोर करती है।

महत्वपूर्ण सेटिंग्स:

[user]
name = Your Name
email = your.email@example.com

[core]
editor = code --wait
autocrlf = true

[alias]
st = status
co = checkout
br = branch

14.4 .gitkeep फाइल

क्या है .gitkeep?

.gitkeep एक कन्वेंशन है जिसका उपयोग खाली फोल्डर्स को Git में ट्रैक करने के लिए किया जाता है।

उपयोग:

# एक खाली फोल्डर बनाएं
mkdir logs

# .gitkeep फाइल बनाएं
touch logs/.gitkeep

14.5 .gitmodules फाइल

क्या है .gitmodules?

.gitmodules फाइल सबमॉड्यूल्स की कॉन्फिगरेशन इनफॉर्मेशन स्टोर करती है।

उदाहरण:

[submodule "libs/awesome-lib"]
path = libs/awesome-lib
url = https://github.com/user/awesome-lib.git
branch = main

14.6 .git/hooks फोल्डर

क्या हैं Git Hooks?

Git Hooks स्क्रिप्ट्स हैं जो Git कमांड्स के निष्पादन से पहले या बाद में ऑटोमैटिकली चलती हैं।

महत्वपूर्ण Hooks:

  • pre-commit: कमिट से पहले चलता है
  • pre-push: पुश से पहले चलता है
  • post-merge: मर्ज के बाद चलता है

उदाहरण (pre-commit):

#!/bin/sh
# लिंटिंग चेक
npm run lint

# टेस्ट्स रन करें
npm test

याद रखें:

ये कॉन्फिगरेशन फाइल्स आपके Git वर्कफ्लो को सुचारू और व्यवस्थित बनाने में मदद करती हैं। इन्हें प्रोजेक्ट की शुरुआत में ही सेटअप करें।
