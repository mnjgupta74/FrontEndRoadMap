Git Commands Guide for Beginners

This guide will help you understand Git commands and their usage in a team environment.

1. Initial Setup

1.1 Configure Git

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
These commands set up your identity for all Git repositories on your system.

2. Starting a New Project

2.1 Initialize a New Repository

git init

Example:

mkdir my-project

cd my-project

git init

2.2 Clone an Existing Repository

git clone [repository-url]

Example:

git clone https://github.com/username/project-name.git

3. Basic Workflow

3.1 Check Status

git status
Always check status before making changes to see what files are modified.

3.2 Add Files to Staging

git add [file-name]
git add .

Example:

git add index.html

git add . (to add all changed files)

3.3 Commit Changes

git commit -m "Your commit message"
Write clear, descriptive commit messages that explain what changes were made.

4. Working with Branches

4.1 Create a New Branch

git branch [branch-name]
git checkout -b [branch-name]

Example:

git checkout -b feature/login-page

4.2 Switch Between Branches

git checkout [branch-name]
Always commit or stash your changes before switching branches.

4.3 List All Branches

git branch
git branch -a

5. Team Collaboration

5.1 Push Changes to Remote

git push origin [branch-name]

Example:

git push origin feature/login-page

5.2 Pull Latest Changes

git pull origin [branch-name]
Always pull latest changes before starting work on a new feature.

5.3 Resolve Merge Conflicts

When conflicts occur:
  1. Open the conflicted files
  2. Look for conflict markers (<<<<<<<, =======, >>>>>>>)
  3. Choose which changes to keep
  4. Remove conflict markers
  5. Add and commit the resolved files

6. Common Issues and Solutions

6.1 Undo Local Changes

git checkout -- [file-name]
This will discard changes in the working directory.

6.2 Stash Changes

git stash
git stash pop

Example:

git stash (to save changes temporarily)

git stash pop (to restore saved changes)

6.3 View Commit History

git log
git log --oneline

7. Best Practices

  • Always create a new branch for each feature or bug fix
  • Write clear, descriptive commit messages
  • Pull latest changes before starting work
  • Test your changes before pushing
  • Keep your branches up to date with the main branch
  • Delete branches after merging them

8. Common Team Workflow

  1. Pull latest changes: git pull origin main
  2. Create new branch: git checkout -b feature/your-feature
  3. Make changes and commit: git add . and git commit -m "message"
  4. Push changes: git push origin feature/your-feature
  5. Create Pull Request on GitHub/GitLab
  6. After approval, merge to main branch

Remember:

Git is a powerful tool that helps teams work together efficiently. Take time to understand these commands and practice them regularly. When in doubt, always check the status of your repository before making changes.

9. टीम प्रोजेक्ट में Git का उपयोग (Team Project Guide in Hindi)

यह गाइड विशेष रूप से टीम प्रोजेक्ट्स के लिए है, जहां कई डेवलपर्स एक साथ काम कर रहे हैं।

9.1 प्रोजेक्ट सेटअप

git config --global user.name "आपका नाम"
git config --global user.email "आपका.ईमेल@example.com"
सबसे पहले अपनी पहचान सेट करें। यह जरूरी है ताकि टीम को पता चले कि कौन से चेंजेस किसने किए।

9.2 डेली वर्कफ्लो

सुबह का काम:

git checkout main
git pull origin main

हर दिन की शुरुआत मेन ब्रांच को अपडेट करके करें

नए फीचर पर काम शुरू करना:

git checkout -b feature/your-feature-name

उदाहरण: git checkout -b feature/login-page

9.3 कोड चेंजेस का प्रबंधन

छोटे-छोटे कमिट्स करें:

git add .
git commit -m "लॉगिन फॉर्म वैलिडेशन जोड़ा"
git push origin feature/your-feature-name

9.4 टीम के साथ सिंक रहना

git checkout main
git pull origin main
git checkout feature/your-feature-name
git merge main

रेगुलर इंटरवल पर मेन ब्रांच से अपडेट लेते रहें

9.5 महत्वपूर्ण टिप्स

  • हमेशा नई फीचर के लिए नई ब्रांच बनाएं
  • कमिट मैसेज हिंदी या अंग्रेजी में स्पष्ट लिखें
  • पुश करने से पहले लोकल टेस्टिंग करें
  • टीम के साथ रेगुलर कम्युनिकेशन रखें
  • कॉन्फ्लिक्ट्स को तुरंत रिजॉल्व करें

9.6 कॉन्फ्लिक्ट रिजॉल्यूशन

कॉन्फ्लिक्ट आने पर:

  1. कॉन्फ्लिक्ट वाली फाइल्स चेक करें
  2. टीम से डिस्कस करें
  3. सही कोड रखें और कॉन्फ्लिक्ट मार्कर्स हटाएं
  4. चेंजेस को कमिट करें

9.7 सुरक्षा टिप्स

  • कभी भी सीधे मेन ब्रांच पर काम न करें
  • पासवर्ड या API कीज़ कभी कमिट न करें
  • रेगुलर बैकअप लेते रहें
  • अनजान कमांड्स का उपयोग न करें

याद रखें:

टीम प्रोजेक्ट में Git का सफल उपयोग कम्युनिकेशन, ऑर्गनाइजेशन और डिसिप्लिन पर निर्भर करता है। रेगुलर प्रैक्टिस से आप और आपकी टीम बेहतर तरीके से काम कर सकेंगे।

11. Git Cherry-Pick (विस्तृत गाइड)

Git cherry-pick एक शक्तिशाली कमांड है जो आपको किसी एक कमिट को एक ब्रांच से दूसरी ब्रांच में ले जाने की अनुमति देता है।

11.1 Cherry-Pick क्या है?

Cherry-pick का मतलब है किसी एक कमिट को चुनकर उसे दूसरी ब्रांच में ले जाना। यह तब उपयोगी होता है जब:

  • आप किसी एक फीचर को एक ब्रांच से दूसरी ब्रांच में ले जाना चाहते हैं
  • किसी बग फिक्स को एक ब्रांच से दूसरी ब्रांच में ले जाना चाहते हैं
  • किसी एक कमिट को मर्ज करना चाहते हैं, पूरी ब्रांच को नहीं

11.2 Cherry-Pick का उपयोग कैसे करें?

बेसिक सिंटैक्स:

git cherry-pick <commit-hash>

उदाहरण:

मान लीजिए आपके पास दो ब्रांच हैं:

  • main (मुख्य ब्रांच)
  • feature/login (लॉगिन फीचर ब्रांच)

आप feature/login ब्रांच में एक बग फिक्स किया है और उसे main ब्रांच में ले जाना चाहते हैं:

# पहले feature/login ब्रांच में जाएं
git checkout feature/login

# कमिट हिस्ट्री देखें
git log --oneline

# मान लीजिए बग फिक्स का कमिट हैश है: abc1234
# main ब्रांच पर स्विच करें
git checkout main

# कमिट को cherry-pick करें
git cherry-pick abc1234

11.3 Cherry-Pick के विकल्प

1. कमिट को एडिट करके cherry-pick करना:

git cherry-pick -e abc1234

2. कमिट मैसेज को बदले बिना cherry-pick करना:

git cherry-pick -x abc1234

3. कई कमिट्स को एक साथ cherry-pick करना:

git cherry-pick abc1234 def5678

11.4 Cherry-Pick केस स्टडी

सीनारियो: आपके पास एक वेबसाइट प्रोजेक्ट है जिसमें दो ब्रांच हैं:

  • main (प्रोडक्शन कोड)
  • staging (टेस्टिंग के लिए)

समस्या: staging ब्रांच में एक सुरक्षा फिक्स किया गया है जिसे तुरंत main ब्रांच में ले जाना है।

समाधान:

# staging ब्रांच में जाएं
git checkout staging

# कमिट हिस्ट्री देखें
git log --oneline

# सुरक्षा फिक्स का कमिट हैश है: 789xyz
# main ब्रांच पर स्विच करें
git checkout main

# कमिट को cherry-pick करें
git cherry-pick 789xyz

# चेंजेस को रिमोट में पुश करें
git push origin main

11.5 Cherry-Pick के फायदे

  • स्पेसिफिक चेंजेस को टारगेट कर सकते हैं
  • पूरी ब्रांच को मर्ज किए बिना एक कमिट ले सकते हैं
  • क्विक फिक्स को एक ब्रांच से दूसरी ब्रांच में ले जा सकते हैं
  • फ्लेक्सिबल वर्कफ्लो बनाता है

11.6 Cherry-Pick के नुकसान

  • कॉन्फ्लिक्ट्स हो सकते हैं
  • कमिट हिस्ट्री कॉम्प्लेक्स हो सकती है
  • डिपेंडेंसी इश्यूज हो सकते हैं
  • टेस्टिंग जरूरी है

11.7 बेस्ट प्रैक्टिसेज

  • हमेशा पहले टेस्ट करें
  • कॉन्फ्लिक्ट्स को तुरंत रिजॉल्व करें
  • क्लियर कमिट मैसेज रखें
  • डिपेंडेंसीज को ध्यान में रखें
  • डॉक्युमेंटेशन अपडेट करें

Remember:

Git is a powerful tool that helps teams work together efficiently. Take time to understand these commands and practice them regularly. When in doubt, always check the status of your repository before making changes.
